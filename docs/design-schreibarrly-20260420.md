## Problem Statement

Non-native professionals writing in German at work have no tool that handles context-aware correction. They write emails, Jira comments, Miro notes, and Teams messages — each with different register, length, and tone requirements. Today they copy-paste text into ChatGPT with manual context tags like "this is an email to my boss / this is just a Jira comment" before getting corrections. This is 6–7 steps of friction for something they do dozens of times a day.

The problem is not just grammar. Technically correct German can still be wrong for the context — too casual for a client email, too formal for a Teams chat, too long for a Jira ticket. No existing tool (LanguageTool, DeepL Write, Grammarly) understands context. None of them remember your patterns to help you improve over time.

## What Makes This Cool

- **Zero-config context detection**: the app knows you're in Jira vs. Outlook vs. Teams from the active window title. You never tell it what you're writing — it already knows.
- **Completely offline and private**: runs on a local model (Ollama). Your text never leaves your machine.
- **The teaching session**: every correction is logged. Once a week, you get a summary of your recurring mistakes. You're not just fixing today's message — you're actually getting better at German.

## Constraints

- Windows desktop (target users write in Outlook, Teams desktop, Miro desktop, Chrome-based Jira)
- Free at launch (open source, local model = zero API costs)
- Must be usable by non-technical teammates (installable without a developer setup)
- Privacy-first — no text sent to external servers in the local-model version

## Premises

1. Windows desktop is the primary target — users write in Outlook, Teams, Miro, and browser-based Jira.
2. Context detection is the core differentiator — auto-detecting Jira vs. email vs. Teams is what separates this from a smarter clipboard paste.
3. The weekly teaching session is the long-term stickiness — grammar correction alone is a commodity.
4. Target users are already comfortable copy-pasting into ChatGPT — a faster, private, context-aware desktop tool is a natural upgrade.

## Cross-Model Perspective

Independent Claude subagent cold read — key findings:

**Coolest version not yet considered:** A clipboard interceptor where the corrected text is already on the clipboard by the time you press Ctrl+V. The correction happens in the background the moment you copy. Zero new habit. Tray icon goes amber while thinking, green when ready.

**Key quote that reveals the emotional core:** "it learns my mistakes and gives me a language teaching session once a week" — This is someone who feels embarrassed about their German. The copy-paste workflow is a daily reminder of not being fluent. The teaching session is emotional, not functional: "I want to stop needing this tool eventually." Positioning: the writing tool that makes itself unnecessary. That's a loyalty hook most tools would never voluntarily build.

**50% already exists:** `pygetwindow` + `pyperclip` + a lightweight LLM call. `pygetwindow` gives you active window title in two lines. `pyperclip` handles clipboard. The missing 50%: mistake tracker (SQLite), weekly session UI, window-to-context mapping (maintenance-heavy), and packaging for non-technical users (PyInstaller + pystray is painful).

## Approaches Considered

### Approach A: Cloud API (GPT-4o-mini / Claude Haiku)
- Best German quality (~GPT-4o-mini level)
- Costs ~€0.02/100 corrections — nearly free at individual scale
- Requires API key setup; text leaves user's machine
- Enables future paid tier (free 30 corrections/month, then €4/month)
- Risk: privacy-conscious users won't try it

### Approach B: Local Model via Ollama (Recommended for V1)
- Completely offline — no text leaves the machine
- Zero cost forever
- German quality: ~80% of GPT-4o-mini, good enough for Jira/Teams
- Ollama installation required (one-time, ~5 minutes)
- Open source — easy to post to GitHub and expat communities
- Risk: lower correction quality for complex formal writing (client emails)

### Approach C: Hybrid
- Local model by default; user can optionally add an API key for cloud quality on formal emails
- Best of both, but more complex to build and explain
- V2, not V1

## Recommended Approach

**Approach B: Local model, open source, validate first.**

Build the local version. Publish on GitHub. Post to expat communities (r/germany, r/expats, German expat Slack groups, Germany-based professional forums). Measure: stars, issues filed, people asking for the cloud version. If interest exists, build the paid cloud version. If not — you've still built something that solves your own problem, for free, forever.

The cloud version is the business. The local version is the bet that there's a market.

## V1 User Workflow

1. User writes German text in any app (Jira, Teams, Outlook, Miro)
2. User selects all text → **Ctrl+C** (normal copy)
3. User presses **Ctrl+Shift+V** (the Schreibarrly hotkey)
4. App reads clipboard + active window title → detects context
5. Sends text to Ollama with context-specific system prompt
6. Corrected German replaces clipboard content (~2–10 seconds depending on hardware)
7. Tray icon: amber while processing, green when ready
8. User presses **Ctrl+V** → corrected text appears in the original app
9. Correction logged to SQLite (original + corrected + context + timestamp)

**V1 scope:** hotkey trigger only. No auto-intercept on Ctrl+C (that's V2 if users want it).

**V1 logging scope:** log every correction to SQLite immediately (original + raw_response + corrected + context + timestamp). The `raw_response` column enables LLM evals — compare what Ollama returned vs. what was used, measure preamble-stripping correctness over time. The weekly report UI is V2.

**Hotkey:** `Ctrl+Shift+V` is "paste as plain text" in Teams and Chrome. Use `Ctrl+Shift+G` as default (no known conflicts in Outlook, Teams, Chrome, or Miro on Windows). Hotkey is user-configurable in `config.json` (`"hotkey": "<ctrl>+<shift>+g"`).

**Hotkey library:** `pynput` — no admin rights required, works in corporate environments without elevated permissions.

**Single-instance:** Acquire a named Windows mutex (`SchreibarrlyMutex`) on startup. If already held, show notification "Schreibarrly is already running." and exit.

**First-run experience:** On first launch (detected by absence of `%APPDATA%\Schreibarrly\config.json`), show a single toast notification after the tray icon appears:
> "Schreibarrly is running! Press Ctrl+Shift+G to correct text."

This toast fires once only — the act of creating `config.json` (which happens on first run during startup initialization) marks first-run as complete. No persistent "first-run flag" needed.

**Correction UX flow:**
1. Hotkey fires → immediately show toast: "Correcting: [first 40 chars of clipboard]..."
2. Tray → amber
3. Ollama call (with threading lock — concurrent hotkey press shows "correction in progress" and does nothing)
4. Tray → green
5. Clipboard updated silently. User presses Ctrl+V.

**UI language: English throughout** (tray menu, notifications, all copy). The corrected German text is produced by the LLM — the app interface itself is in English, consistent with non-native speakers' work environment.

**Tray context menu (right-click on tray icon):**
```
Schreibarrly — idle            [status line, not clickable, updates with app state]
────────────────────────────
About Schreibarrly             [shows version toast: "Schreibarrly v0.1.0 — Hotkey: Ctrl+Shift+G"]
────────────────────────────
Quit                           [exits the process cleanly, releases mutex]
```
Status line text states:
- Idle: "Schreibarrly — idle"
- Processing: "Schreibarrly — correcting..."
- Error: "Schreibarrly — error (Ollama not running)"

Tray icon hover tooltip: "Schreibarrly — idle" (mirrors status line).

"About Schreibarrly" shows a toast: "Schreibarrly v0.1.0 — Hotkey: Ctrl+Shift+G"
This serves double duty: version info + hotkey reminder for users who missed the first-run toast.

Config changes in V1 require editing `%APPDATA%\Schreibarrly\config.json` directly (documented in README). Settings UI is V2.

**Error states:**
- Ollama not running → tray icon red, notification: "Schreibarrly: Ollama is not running. Start Ollama and try again."
- Ollama timeout (>30s, configurable) → tray icon red, notification: "Schreibarrly: correction timed out. Try again."
- LLM response contains preamble → strip known preamble phrases ("Hier ist", "Natürlich", "Gerne", "Der korrigierte Text") through first colon or newline. Otherwise return as-is. Raw response always logged.
- No active window when hotkey fires → use "default" context; do not crash.
- Clipboard is empty or contains non-text (image, file) → notification "Schreibarrly: no text in clipboard." Take no action.

## Context Detection Rules (V1)

**Window detection implementation:** Use `pywin32` (`win32gui.GetForegroundWindow()` + `win32process.GetWindowThreadProcessId()` + `psutil.Process(pid).name()`) rather than `pygetwindow` (unmaintained since 2019). This gives both window title and process name, which are both needed for reliable context matching.

Window title pattern matching (case-insensitive substring):

Rules checked **top-to-bottom, first match wins** (case-insensitive):

| Priority | Context | Process + title pattern | System prompt behavior |
|---|---|---|---|
| 1 | Teams | `Teams.exe` OR `ms-teams.exe` OR (chrome/msedge + "Microsoft Teams") | Casual but professional. Natural German. Keep it short. |
| 2 | Outlook | `OUTLOOK.EXE` | Formal register. Full sentences. Appropriate closing. |
| 3 | Jira | chrome/msedge + "Jira" OR "Atlassian" | Concise, action-oriented, no filler. Max 3 sentences. |
| 4 | Miro | `Miro.exe` OR chrome/msedge + "Miro" | Short sticky-note style. Direct. Conversational ok. |
| 5 | Default | anything else | Fix grammar and phrasing. Preserve register and length. |

Note: `ms-teams.exe` is the Teams 2.0 process name on Windows 11 (shipped 2023). Both must be matched.

System prompt template (filled per context):
```
Du bist ein professioneller Lektor für Deutsch als Fremdsprache.
Kontext: [CONTEXT_DESCRIPTION]
Aufgabe: Korrigiere den folgenden Text. Behalte die ursprüngliche Bedeutung.
Korrigiere: Grammatik, Satzbau, Groß-/Kleinschreibung, Kommasetzung.
Passe an: Register und Stil für den angegebenen Kontext.
Antworte NUR mit dem korrigierten Text. Keine Erklärungen.
```

## Ollama API Call Spec

Endpoint: `http://localhost:11434/api/generate` (configurable in `config.json`)
Method: POST, blocking (not streaming)
Timeout: 30 seconds (configurable in `config.json` as `"timeout_seconds"`)
Model parameter: configurable in `config.json`, default `"marco/em_german_mistral_v01"`
HTTP: use a persistent `requests.Session()` initialized at startup for connection keep-alive

Request body:
```json
{
  "model": "<model_from_config>",
  "prompt": "<system_prompt>\n\nText:\n<clipboard_content>",
  "stream": false
}
```

Response: parse `response.json()["response"]` and strip leading/trailing whitespace.

DB path: `%APPDATA%\Schreibarrly\corrections.db` (directory + file created on first run if not present).
Config path: `%APPDATA%\Schreibarrly\config.json` (never next to the .exe — Program Files is read-only without admin).

## Mistake Logger (V1 — lightweight)

SQLite tables:
```sql
CREATE TABLE schema_version (
  version INTEGER PRIMARY KEY
);
INSERT INTO schema_version VALUES (1);  -- bump on schema changes

CREATE TABLE corrections (
  id INTEGER PRIMARY KEY,
  ts DATETIME DEFAULT CURRENT_TIMESTAMP,
  context TEXT,          -- 'jira', 'teams', 'outlook', 'miro', 'default'
  original TEXT,
  raw_response TEXT,     -- unmodified Ollama output, for evals
  corrected TEXT         -- after preamble stripping
);
```

On startup: check `schema_version`. If missing, create schema at version 1. Future versions add `ALTER TABLE` migrations — never drop and recreate.

Diff extraction for the weekly report: use `difflib.ndiff()` at word level. Group recurring patterns by changed word/phrase (e.g. "das" → "dass" appearing 7 times). Weekly report = local HTML file opened in browser via `webbrowser.open()`. V1 is manual trigger ("generate my weekly report"), not automated.

**Teaching session is V1 data-only** (log every correction). The weekly report UI is V2.

## Open Questions

- Which local German model performs best for register-aware correction? (Mistral 7B, Llama 3.1 8B, or a German-fine-tuned variant like EM German Mistral — test all three)
- Ollama cold-start latency on mid-range hardware: first inference on a 7B model can be 3–10 seconds. Show a progress indicator in the tray icon. Document expected latency in README.
- Window title localization: German Windows may show different process/title strings for Outlook etc. Test on a German-locale Windows installation.
